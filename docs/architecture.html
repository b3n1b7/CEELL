CEELL

Configurable ECU Execution Layer for 48V Vehicle Domain Controllers

Think of CEELL as a "smart brain" that lives inside each electronic controller in a vehicle. Every controller runs the same software — it reads a config file to know its job (body, chassis, lights...) and then automatically finds the other controllers on the network. Write once, deploy everywhere — even on different chip vendors.

The Big Picture
From Silicon to Software — What Lives Where
A vehicle ECU is like a layered cake. At the bottom: the physical chip and wires. At the top: the application logic. We are building the middle layers — the middleware that makes ECUs configurable, discoverable, and vendor-independent. Everything below our code already exists (Zephyr, chip vendors). Everything above is what vehicle engineers will configure.
Why This Matters
One Codebase, Many Chips
Because we build on Zephyr (which supports 600+ boards), our CEELL code doesn't care what chip is underneath. Switching from NXP to ST means changing one line in the build command — not rewriting the application. This is the same strategy used by Tesla (moving from Intel to AMD) and Apple (Intel to ARM): keep the software portable, swap the hardware.

Only the bottom two layers change when you switch chips.

Our CEELL code and Zephyr stay identical. Moving from NXP to ST means: west build -b st_board_name instead of west build -b mr_canhubk3. The discovery, config, networking — all the logic — stays exactly the same. That's the whole point of building on Zephyr: hardware independence.

Phase 1 — What We Built
3 Virtual ECUs Discovering Each Other
In simulation, three virtual ECUs boot up and automatically find each other using multicast messages over a virtual Ethernet switch. Each ECU announces "I'm here, my role is X" once per second. On real hardware, these same messages would travel over automotive Ethernet cables between physical boards.
How It Works
Nodes Find Each Other Automatically
Every second, each node broadcasts a message saying "Hi, I'm node X, my role is Y." When a node hears a message from someone new, it adds them to its list of known peers. Within about 1 second of booting, every node knows about every other node. No manual configuration needed — just plug in and go.
Inside Each Node
What Happens When a Node Powers On
Each ECU follows the same 5-step startup sequence. The whole process takes milliseconds. By the end, the node is on the network and talking to its peers.
STEP 1
Read Configuration
The node reads a JSON file from its flash memory to learn its identity — "Am I the body controller? The lights controller?" If flash is empty, it uses built-in defaults.
STEP 2
Lock In Identity
The node stores its ID, role, name, and IP address in memory. This can't be changed while running — the node knows who it is for the rest of its life.
STEP 3
Open Network Socket
Opens a communication channel on the Ethernet network and joins a "multicast group" — think of it as tuning into a shared radio frequency that all CEELL nodes listen to.
STEP 4
Start Discovery
Two background workers start: one broadcasts "I'm here" every second, the other listens for announcements from other nodes and adds them to a peer list.
STEP 5
Health Monitoring
Every 5 seconds, the node logs a health check: "I'm alive, I know about N peers." This runs forever and is the heartbeat of the system.

Quality Assurance
Automated Testing Pipeline
Every time code is pushed to GitHub, an automated pipeline builds all 3 firmware variants and runs them in a virtual chip simulator (Renode). The tests check that nodes boot correctly and discover each other. If anything breaks, we know immediately.
